# Cloud - Software Testing

## 1. Smoke Test

```
1. Smoke Test : BVT, Build Vertification Testing이라고도 하며 본격적인 테스트의 수행에 앞서 시스템, 컴포넌트, 소프트웨어 프로그램 등 테스트 대상이나 제품의 빌드가 구축된 테스트 환경에서 테스트가 가능한지를 판단하기 위해 주요 모듈이나 시스템을 간단하게 테스트 하는 것

2. 목적 : Smoke test는 특정 빌드마다 수행하거나 매 빌드마다 각각 수행될 수 있다. 만약 해당 빌드가 리소스를 투입해서 테스트를 진행할 가치가 있는 빌드인지 판단하고 개발팀에 피드백을 주기 위한 목적을 

3. TCL, Test Case List : 테스팅을 위해 다수의 기능 군 영역, 품질 속성 영역으로 나누어 작성된 목록이다. TCL의 각 기능 군마다 1-4개 정도의 케이스를 추려서 하나의 양식으로 정리하면 Smoke Test 수행을 위한 테스트 케이스가 완성된다

4. Smoke Test 완료 이후 업무 처리 : Smoke Test 내용에 대해서 Peer Review(동료 간 검토)를 통해 feedback이 이루어 진다. 이런 리뷰가 2회 이상이 진행되면 QA 레벨에서 합의된 결과물이 도출되게 된다.

5. 협의가 완료된 이후 업무 처리 : 확정된 내용을 개발팀 전체를 대상으로 공표하고 Smoke Test가 필요하다고 QA에서 판단하거나 개발팀의 요청이 있는 경우 빌드를 선정해 Smoke Test를 수행하고 해당 결과를 분석해 개발팀에 통보하게 된다

6. Smokde Test를 기준으로 삼는 이슈들
	1) 모든 테스트의 시작에는 Smoke Test가 반드시 있어야 한다. TCL test를 시작하려면 Smoke Test를 통과된 빌드를 바탕으로 한다. 
	2) 버그 추적 시스템에 올라온 이슈는 Smoke Test를 통과한 빌드를 바탕으로 재확인해야 한다. 
	
7. 이외 문제들
	1) Smoke test 내용에 대해서 트집을 잡는 것에 주의해야 한다 : Smoke Test 요소에 대해선 개발팀 및 PM 과 협의를 거쳐야 한다.
	2) Smoke Test에 대해서 가급적 테스트하는 사람에 따라 결과가 달라질 수 있는 요소는 배제해야 한다.
	3) Test 허들의 문제 : 어느 수준에 이른 빌드는 Smoke Test를 가볍게 통과해버린다. 따라서 진정한 테스트를 위해 Smoke Test 요소의 변경, 추가 및 삭제를 고려할 필요도 있다.
```
